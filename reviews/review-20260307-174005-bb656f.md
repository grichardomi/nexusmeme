## Changes Reviewed

### Fix 1: Wrong table name in `recordPerformanceFee` (CRITICAL)

**File:** `src/services/billing/performance-fee.ts:47-52`

**Before:**
```sql
LEFT JOIN user_subscriptions us ON u.id = us.user_id
...us.plan,
```

**After:**
```sql
LEFT JOIN subscriptions us ON u.id = us.user_id
...us.plan_tier AS plan,
```

**Assessment: CORRECT.**
- The table `user_subscriptions` does not exist; the real table is `subscriptions` with column `plan_tier`.
- The old code caused PostgreSQL to throw `relation "user_subscriptions" does not exist` on every `recordPerformanceFee` call.
- The error was silently swallowed in the trade close route's catch block, resulting in zero fee records for ~295 profitable trades (Feb 9 → Mar 7).
- Fix is minimal and surgical — no over-engineering.

**No issues found.**

### Fix 2: Dead `else if` branch removed from trade close route

**File:** `src/app/api/bots/trades/close/route.ts:506-513`

**Before:**
```typescript
if (actualProfitLoss > 0) {
  await recordPerformanceFee(...)
} else if (isPaperTrading && actualProfitLoss > 0) {  // unreachable
  logger.info('Skipping fee recording...')
}
```

**After:**
```typescript
if (actualProfitLoss > 0) {
  await recordPerformanceFee(...)
}
```

**Assessment: CORRECT.**
- The `else if` condition (`isPaperTrading && actualProfitLoss > 0`) can never be true when the preceding `if (actualProfitLoss > 0)` was false — a logical contradiction.
- Removing it has no behavioural change; waiving for paper/trial is correctly handled inside `recordPerformanceFee`.
- `tradingMode` added to the success log for observability — minor improvement, no risk.

**No issues found.**

### Admin Cron Dashboard (`src/app/admin/dashboard/page.tsx`)

**Assessment: CORRECT.**
- Cron jobs table shows Interval, Last Run, Status, Next Run per job.
- Schedule is editable via a 16-option preset dropdown + freeform custom cron input; `Next Run` recomputes instantly on change.
- Enable/disable toggle disables Run Now button and dims the row.
- `CRON_SECRET` is never exposed to the client — trigger calls internal proxy route.

**Minor observations (non-blocking):**
- `computeNextRun` is a simplified parser — handles `*/N`, daily, and monthly patterns but not DOW (`0 9 * * 1-5`). Weekday presets will show an approximate next run. Acceptable for an admin dashboard display.
- Job enabled/disabled state is client-side only (not persisted to DB). On page refresh, all jobs reset to enabled. Low risk for admin tooling.

## Recommendations

1. **Backfill 295 missing fee records** — the waived fees from Feb 9→Mar 7 are absent from the dashboard. Run the backfill INSERT if audit completeness matters.
2. **Persist cron enabled/disabled state** — currently resets on refresh. Consider storing in `localStorage` or a DB settings table if admins need persistent disabling.
3. **`computeNextRun` DOW support** — the weekday presets (`Mon @ 09:00`, `Weekdays @ 09:00`) will display imprecise next-run times. Add DOW parsing if accuracy is needed.
