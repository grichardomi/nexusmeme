## Summary

## Issues Found

### FIXED in this session

- Fixed: added `USDC_CHAIN_ID=8453` to `.env.local`, `environment.ts` schema, default env, and API response

### REMAINING ISSUES

**[MEDIUM] Webhook confirmation logic may silently drop payments**
- `usdc/webhook/route.ts:77-88`: `confirmations = eventBlock - txBlock`
- Alchemy ADDRESS_ACTIVITY fires once per confirmation, so `event.event.blockNum` and `activity.blockNum` will differ by at most 1
- With `USDC_REQUIRED_CONFIRMATIONS=1` (current `.env.local`) this works fine
- If `USDC_REQUIRED_CONFIRMATIONS` is raised above 1, payments will be silently skipped (logged as "below threshold") with no retry mechanism
- Recommendation: If higher confirmation threshold is needed, implement a pending-confirmation queue instead of relying on the block delta in a single webhook call

**[MEDIUM] No MetaMask / Web3 wallet connect UI**
- The task mentions "metamask alchemy integration" but no browser-side wallet connect exists
- Current flow is manual: copy address, copy amount, send from external wallet
- The EIP-681 QR code improves this for mobile wallets but desktop MetaMask users have no direct "Pay Now" button
- Recommendation: Add a `window.ethereum` connect button in `USDCPayButton.tsx` that calls `eth_sendTransaction` with the USDC transfer calldata directly from the browser. Alchemy SDK can be used for RPC calls.

**[LOW] Raw USDC value parsing — usdc-payment.ts:208-210**
- Hex path: `BigInt(transfer.value).toString()` — correct, preserves full precision
- Decimal path: `String(Math.round(parseFloat(transfer.value)))` — `parseFloat` loses precision for large values (>2^53)
- For USDC amounts users pay (typically < $10,000 = 10,000,000,000 raw units), this is safe in practice
- Recommendation: Use `BigInt(Math.round(parseFloat(transfer.value))).toString()` or better, require Alchemy to always send hex and remove the decimal path

**[LOW] PerformanceFeesSummary overdueAmount uses billed_fees as proxy**
- `PerformanceFeesSummary.tsx:143`: `const overdueAmount = summary.billed_fees`
- Comment says "Billed but not paid = overdue" but `billed_fees` actually includes all billed fees ever (paid + unpaid)
- If some billed fees were paid via Coinbase Commerce, the overdue banner may show inflated amount
- Recommendation: API should return a dedicated `overdue_amount` field from the DB query rather than using `billed_fees` as a proxy

**[LOW] GoLiveWizard: 500ms setTimeout between stop and switch**
- `GoLiveWizard.tsx:106`: `await new Promise(r => setTimeout(r, 500))`
- Artificial delay for "clean shutdown" — bot stopping is handled server-side, client shouldn't need to wait
- Low risk but adds latency to the go-live flow for no reliable benefit

## Recommendations

1. **Add MetaMask connect flow** to `USDCPayButton.tsx`: detect `window.ethereum`, connect wallet, build USDC `transfer()` calldata, send transaction. This is the core "MetaMask integration" gap.

2. **Confirmation retry queue**: if `USDC_REQUIRED_CONFIRMATIONS > 1` is ever needed, store unconfirmed transfers in a `usdc_pending_confirmations` table and process on each subsequent webhook call.

3. **Fix `overdue_amount` calculation** in the performance fees API to use `SUM WHERE status = 'billed' AND paid_at IS NULL` rather than total billed.

4. **Add `.env.local` documentation** for all `USDC_*` and `ALCHEMY_*` vars in `RAILWAY_DEPLOYMENT.md` per project convention.
